## 3. Taxonomy Enforcement Rules

### Rule 1: No New Categories Outside Enums

Any new categorical value (pattern, modifier, environment, etc.) MUST be added to
`engine/enums.py` first. Code that introduces string literals for categories without
enum backing is a violation.

**Current Violations Found**:

| Location | Violation | Severity |
|----------|-----------|----------|
| `engine/patterns.py:classify_lighting_pattern()` | Returns string literals ("triangle", "clamshell", etc.) not enum values | LOW -- values match enum, but type is `str` not `LightingPattern` |
| `engine/cue_inference.py:_infer_shadow_pattern()` | Returns "butterfly", "flat" as plain strings | LOW -- same pattern |
| `api/routes/shoot_match.py:MOOD_MAP` | Defines mood values as inline strings, not sourced from enums | MEDIUM -- potential drift |
| `ui/src/transform.js:detectLightingPattern()` | JS-side pattern detection using hardcoded strings | MEDIUM -- must stay synced with engine enums |
| `engine/vlw_reconciliation.py:VLM_LIGHTING_STYLE_MAP` | Large mapping table of string literals | LOW -- deliberate mapping from VLM free-text |

### Rule 2: Taxonomy YAML Must Match Enums

Pattern IDs in `data/taxonomy/lighting_patterns.yaml` must correspond 1:1 to
`LightingPattern` enum values. Currently:

- YAML has `flat_frontal`, `hard_frontal`, `side_key`, `top_light_dominant`, `rim_dominant` -- these do NOT have matching `LightingPattern` enum values
- Enum has `flat_fashion`, `bare_bulb_editorial`, `strip_dramatic` -- these do NOT appear in the YAML

**Recommendation**: Reconcile. The YAML represents photographer-facing pattern definitions; the enum represents engine-internal values. They should be aligned with an explicit mapping if they cannot be 1:1.

### Rule 3: UI Must Consume Enum Labels, Not Define Its Own

The UI (`ui/src/transform.js`, `ui/src/data/masterLighting.js`) contains pattern
label mappings that could drift from `engine/enums.py`. The UI should receive label
strings from the API, not maintain its own mapping tables.

**Current State**: Partially compliant. `transform.js` uses `modifierLabel()` with
its own hardcoded map. The API does not consistently send `label` alongside `value`.

## 4. Dataset Locations & Mixing Rules

### Canonical Data (versioned, reviewed)

| Path | Type | Mutable? |
|------|------|----------|
| `data/systems/canonical/*.yml` | Pattern definitions | Only via reviewed PR |
| `data/taxonomy/*.yaml` | Taxonomy definitions | Only via reviewed PR |
| `data/lighting_systems.json` | Generated system database | Only via `generate_systems.py` |
| `data/lighting_patterns.json` | Pattern matching rules | Only via reviewed PR |

### Reference Dataset (versioned, curated)

| Path | Type | Mutable? |
|------|------|----------|
| `data/reference_dataset/` | Curated reference images + analysis | Only via LAB pipeline |
| `data/reference_dataset/_version.json` | Dataset version tracking | Auto-updated |

### Runtime / Generated (NOT canonical)

| Path | Type | Notes |
|------|------|-------|
| `static/uploads/` | User-uploaded reference images | Ephemeral, per-session |
| `__pycache__/` | Python bytecode | Generated |
| `data/mp_models/` | MediaPipe model files | Downloaded dependencies |

### Rule: No Mixing

- Runtime uploads (`static/uploads/`) must NEVER be written into `data/reference_dataset/`
  without going through the LAB ingestion pipeline
- Generated analysis artifacts (debug overlays, signals.json) must be marked with
  processing timestamps and version info
- Test fixtures must not reference runtime-generated data
